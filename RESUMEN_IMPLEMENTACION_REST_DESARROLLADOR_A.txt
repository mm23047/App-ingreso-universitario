   │  └─ POST   /areas/{idArea}/temas           → Crear tema en área

   │  ├─ AreasConocimientoDAO
   │  └─ TemaDAO
   │

   │  ├─ nombreArea obligatorio
   │  ├─ nombreArea único (case-insensitive)
   │  └─ No eliminar si tiene temas asociados (409 CONFLICT)
   │
   └─ Respuestas:
      ├─ 200 OK
      ├─ 201 CREATED (con Location header)
      ├─ 204 NO_CONTENT (DELETE)
      ├─ 400 BAD_REQUEST (validación)
      ├─ 404 NOT_FOUND

      └─ 500 INTERNAL_SERVER_ERROR (con SERVER_EXCEPTION header)

ESTRUCTURA Y PATRONES IMPLEMENTADOS
================================================================================

1. CLASE BASE GENÉRICA
   ├─ AbstractResource<T>
   │  ├─ Método genérico findRange(first, max) para paginación
   │  ├─ Retorna List<T> + header Total-records
   │  └─ Manejo de excepciones centralizado
   │
   └─ Reutilizada por todos los 10 Resources

2. COHERENCIA REST-DAO
   ├─ Cada Resource inyecta su correspondiente DAO via @Inject
   ├─ El DAO contiene la lógica de validación y persistencia
   ├─ El Resource maneja HTTP (códigos, headers, URIs, serialización)
   ├─ Separación clara de responsabilidades
   └─ Las validaciones de negocio viven en el DAO

3. HEADERS HTTP COHERENTES (RestHeaders)
   ├─ Total-records: Cantidad total de registros (paginación)
   ├─ Missing-parameter: Parámetro requerido faltante
   ├─ Not-found-id: ID que no fue encontrado
   ├─ Server-exception: Descripción de error de servidor
   └─ Conflict-reason: Razón del conflicto de negocio

4. CÓDIGOS HTTP COHERENTES
   ├─ 200 OK: GET exitoso, PUT exitoso
   ├─ 201 CREATED: POST exitoso (con Location header)
   ├─ 204 NO_CONTENT: DELETE exitoso
   ├─ 400 BAD_REQUEST: Validación de entrada (parámetro obligatorio)
   ├─ 404 NOT_FOUND: Entidad no existe
   ├─ 409 CONFLICT: Validación de negocio (duplicado, dependencias)
   └─ 500 INTERNAL_SERVER_ERROR: Error en la aplicación

5. CONVENCIONES DE URI RELACIONAL
   ├─ GET    /recursos                    → Listar todos
   ├─ POST   /recursos                    → Crear
   ├─ GET    /recursos/{id}               → Obtener uno
   ├─ PUT    /recursos/{id}               → Actualizar
   ├─ DELETE /recursos/{id}               → Eliminar
   ├─ GET    /recursos/{id}/subrec        → Listar sub-recursos
   ├─ POST   /recursos/{id}/subrec        → Crear sub-recurso
   │
   └─ Rutas anidadas: Los payloads NO repiten IDs del padre

6. CONVENCIONES DE NAMING
   ├─ Sustantivos en plural: /areas, /temas, /preguntas, /opciones
   ├─ Snake_case para multi-palabras: /respuestas_globales, /pruebas_admision
   ├─ Path params con ID: {idArea}, {idTema}, {idPregunta}, etc.
   ├─ Query params: first, max (paginación)
   └─ MediaType: APPLICATION_JSON para todo

7. MANEJO DE ERRORES CONSISTENTE
   ├─ Todos los endpoints capturan IllegalArgumentException (validación)
   ├─ Todos los endpoints capturan Exception genérica (servidor)
   ├─ Se responde con código HTTP apropiado + header descriptivo
   ├─ Body contiene mensaje legible en español
   └─ Server-exception header contiene detalles técnicos

8. PAGINACIÓN CONSISTENTE
   ├─ Query params: ?first=0&max=50
   ├─ Response headers: Total-records = N
   ├─ Paginación manual para resultados en memoria (findByArea, etc.)
   ├─ Paginación vía DAO.findRange() para queries principales
   └─ Default: first=0, max=50
